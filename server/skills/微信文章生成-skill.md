# 微信公众号文章生成规则

## 目标

把设计思路/技术方案/调研笔记转化为技术读者能快速吸收的微信公众号文章。
**保留技术深度，不稀释概念**——读者是有技术背景的工程师，不需要把专业问题翻译成大白话。

## 输入分流（首要决策）

先判断输入文档的完整度，再决定改写力度。**不要对完整文章套模板重写**。

| 输入类型 | 特征 | 处理策略 |
|---|---|---|
| 碎片化思路 | 列要点、结论模糊、未成文 | 按骨架重构，补全逻辑链，补缺失环节 |
| 半成文 | 有段落但松散、标题层级混乱 | 调整脉络，合并冗余，可视化关键流程 |
| 完整文章 | 结论清楚、结构完整 | **只做必要的结构优化**：调标题层级、补可视化、拆过长段落，不重写正文 |

> **💡 判断信号**
>
> | 信号 | 推断 |
> |---|---|
> | 输入已有 `##` 三级以上标题且语义连贯 | 完整文章 → 走"结构优化"路径 |
> | 输入是列表/短句堆砌 | 碎片化 → 走"重构"路径 |
> | 输入有段落但缺主线 | 半成文 → 走"整理脉络"路径 |

### 完整文章的边界（必须遵守）

对完整文章**禁止**：
- ❌ 扩写正文、补充作者未说的内容
- ❌ 改写结论、调整语义倾向
- ❌ "丰富"已有论点、加"延伸思考"
- ❌ 强行套「概览→分析→总结」三段式

对完整文章**允许**：
- ✅ 修正标题层级（如 `###` → `##`）
- ✅ 为已有流程性描述补 ASCII 流程图
- ✅ 为已有对比性描述补表格
- ✅ 拆分超过 5 行的纯文字段落
- ✅ 合并重复论点、调整章节顺序使脉络更顺

## 文章开头说明（必加）

每篇文章开头必须有一个简短说明（导语），告诉读者：
- **这篇文章介绍什么**：一句话概括文章的主题/解决的问题
- **这篇文章有什么用**：说明读完能获得什么、适合谁看、能用在什么地方

格式模板：

```markdown
> 📖 **本文介绍**
> 【一句话概括主题：解决什么问题 / 分析什么技术 / 总结什么经验】
>
> **适用场景**：【谁适合看：遇到类似问题的工程师 / 某领域从业者】
> **收获预期**：【读完能得到什么：一套排查思路 / 一个技术方案 / 某类问题的解决模板】
> **文章思路**：【整体脉络：① xxx → ② xxx → ③ xxx，按什么顺序展开】
```

示例：

```markdown
> 📖 **本文介绍**
> 分析 Redis 连接池耗尽的根因，给出一套从监控告警 → 日志定位 → 参数调优的完整排查链路。
>
> **适用场景**：遇到 Redis timeout / connection refused 的后端工程师
> **收获预期**：掌握连接池问题的排查方法论，能复用到其他资源池（DB 连接池、线程池）
> **文章思路**：① 问题定位（监控告警 → 日志分析） → ② 根因排查（连接泄漏 / 配置不当） → ③ 解决方案（参数调优 + 代码修复）
```

> **💡 开头说明要点**
>
> | 要点 | 说明 |
> |---|---|
> | 长度 | 导语 ≤ 3 行，不展开细节 |
> | 定位 | 引导读者判断「这篇文章与我相关吗」，而非替代正文 |
> | 格式 | 引用块 + 📖 emoji，与技术性 💡 说明块区分 |

## 文章骨架（按需取用，非固定三段式）

```
## 一、概览（完整文章若已有等价开头则不强制加）
  - 一句话讲清「解决什么问题 / 结论是什么」
  - 整体架构图 / 流程图
  - 关键要点表（≤5 条）

## 二、详细分析（按输入内容切分，不强制 2.1/2.2 编号）
  - 每个子主题独立成章
  - 优先用图表表达，文字补充

## 三、总结与探讨（按需，完整文章若已收尾则不加）
  - 核心收获（3-5 条，每条 ≤20 字）
  - 待探讨问题（可选，2-3 个，说明"为什么值得探讨"）
  - 延伸阅读（可选）
```

## 技术性原则

| 原则 | 说明 | 反例 |
|---|---|---|
| 术语不降级 | 保留"幂等""CAP""WAL""B+ 树""MVCC"等术语，首次出现给一行定义 | 把幂等翻译成"重复请求也返回一样的结果"作为正文 |
| 保留数据/公式 | 复杂度 O(log n)、QPS、p99、字节布局、公式等保留原样 | 把 O(log n) 改成"很快" |
| 代码示例最小化 | 只保留核心片段，≤20 行，去掉 boilerplate | 粘贴完整 main 函数 |
| 决策带理由 | 每个技术选择说明 why，不只说 what | 只写"采用 Redis 分布式锁"不说为什么不用 ZK |
| 边界条件显式化 | 空输入/超时/并发竞争/部分失败等单独列出 | 藏在正文段落里 |
| 取舍显式化 | 列出被放弃的方案及放弃原因 | 只讲选中的方案 |

## 可视化规范

### 必须可视化的场景

| 场景 | 图类型 | 触发条件 |
|---|---|---|
| 多步流程 | 纵向步骤框 | 超过 3 步的逻辑 |
| 分支判断 | 条件框 + 是/否分支 | 有 if/else 语义 |
| 方案对比 | 三列表格 | ≥2 个选项 |
| 调用链路 | 横向箭头图 | 多组件协作 |
| 数据变化 | 前后对比块 | 数据结构/状态迁移 |
| 表结构关系 | 关系图 | 多表外键关联 |

### ASCII 流程图（与下游 md→HTML skill 对齐）

单步骤框：
```
┌─────────────────────────────────────┐
│ ① 步骤标题                           │
│    具体操作描述                       │
└──────────────────┬──────────────────┘
                   ▼
```

分支判断：
```
┌─────────────────────────────────────┐
│ 判断条件?                            │
│    ├─ 是 → 结果 A                   │
│    └─ 否 → 结果 B                   │
└─────────────────────────────────────┘
```

并行时序：
```
请求 A                              请求 B
  │                                   │
  │── 操作1 ──→ 成功 ✓                │── 操作1 ──→ 失败 ✗
  │                                   │
  │   继续                             │   返回错误
```

数据变化对比：
```
操作前:  表名 (字段1=值1, 字段2=值2)    ← 注释
操作后:  表名 (字段1=新值, 字段2=新值)  ← 变化说明
```

> **💡 流程图要点**
>
> | 要点 | 说明 |
> |---|---|
> | 框宽对齐 | 同一图内所有框宽度一致，用 `─` 对齐 |
> | 箭头用 Unicode | `↓` `→` `▼` `↻`，不用 SVG/mermaid |
> | 步骤编号 | 用 ①②③ 标记主流程步骤 |
> | 分支缩进 | `├─` `└─` 标明分支 |
> | 必有图题 | 图前加 `**图 N** 标题` |

## 💡 说明块规范

每个流程图/代码块/表结构后，用 blockquote 格式添加解释，**说明块内禁止超过 5 行纯文字**，超出改表格：

```markdown
> **💡 标题**
>
> | 设计决策 | 为什么 | 如果不这么做 |
> |---|---|---|
> | 决策A | 原因 | 后果 |
```

## 排版规范

### 标题层级

```
## 一、概览        ← 章节标题（中文数字）
### 2.1 主题      ← 子主题（阿拉伯数字）
#### 细节        ← 更细分（少用，≤4 级）
```

### 强调方式

- **重点**：加粗
- `代码/命令/字段名`：行内代码
- > 💡 说明：引用块 + emoji（仅用于解释块）

### 段落约束

| 约束 | 阈值 |
|---|---|
| 纯文字段落上限 | 5 行（超出拆分或改图表） |
| 连续列表项上限 | 5 项（超出需分组或改表格） |
| 标题嵌套上限 | 4 级 |
| 代码片段上限 | 20 行 |
| 表格列数上限 | 4 列（超出拆表） |

### 禁止项

- ❌ 超过 5 行的纯文字段落
- ❌ 连续 3 个以上无分组的列表项
- ❌ 无标题的图表
- ❌ 过深的嵌套（标题不超过 4 级）
- ❌ "参考 xxx 实现" 而不说明差异
- ❌ 伪代码（读者不写 Go/Java，用自然语言流程图）

## 输出契约

1. 只输出 markdown 文章本身，不要任何解释，不要代码围栏包裹整篇
2. 保留输入中的技术准确性，**不编造数据/引用/链接**
3. 对完整文章：不扩写、不删减结论、不改语义，仅调整结构与可视化
4. 对碎片化输入：按骨架补全，补足的逻辑必须可验证、不杜撰
5. 全文无超过 5 行纯文字段落
6. 每个流程/算法都有流程图，每个流程图后都有 💡 解释块
